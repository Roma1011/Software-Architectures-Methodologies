# ფიზიკური საზღვრები (Physical Boundaries)

Bounded context-ები არ არის მხოლოდ **მოდელის საზღვრები** ისინი ასევე წარმოადგენენ **იმ სისტემების ფიზიკურ საზღვრებს**, რომლებიც ამ მოდელებს რეალიზებენ.

ყოველი bounded context უნდა იყოს იმპლემენტირებული როგორც **ცალკე სერვისი ან პროექტი**, რაც ნიშნავს, რომ ის:

- იმპლემენტირდება დამოუკიდებლად
    
- ვითარდება დამოუკიდებლად
    
- ვერსიონდება დამოუკიდებლად
    

სხვა bounded context-ებისგან.

Bounded context-ებს შორის მკაფიო ფიზიკური საზღვრები საშუალებას გვაძლევს **თითოეული bounded context დავაიმპლემენტოთ იმ ტექნოლოგიური სტეკით, რომელიც ყველაზე კარგად შეესაბამება მის საჭიროებებს**.

როგორც ადრე განვიხილეთ, bounded context შეიძლება შეიცავდეს **რამდენიმე subdomain-ს**.

ასეთ შემთხვევაში:

- bounded context წარმოადგენს **ფიზიკურ საზღვარს**
    
- მისი თითოეული subdomain წარმოადგენს **ლოგიკურ საზღვარს**
    

ლოგიკურ საზღვრებს სხვადასხვა პროგრამირების ენაში განსხვავებული სახელები აქვთ, მაგალითად:

- namespaces
    
- modules
    
- packages

# პასუხისმგებლობის საზღვრები (Ownership Boundaries)

კვლევები აჩვენებს, რომ **კარგი ღობეები ნამდვილად ქმნის კარგ მეზობლებს**.

პროგრამული უზრუნველყოფის პროექტებში შეგვიძლია გამოვიყენოთ **მოდელის საზღვრები  bounded context-ები გუნდების მშვიდობიანი თანაარსებობისთვის**.

გუნდებს შორის სამუშაოს განაწილება კიდევ ერთი **სტრატეგიული გადაწყვეტილებაა**, რომელიც შეიძლება მიღებულ იქნას bounded context pattern-ის გამოყენებით.

ერთი bounded context უნდა იყოს:

- იმპლემენტირებული
    
- განვითარებული
    
- შენარჩუნებული
    

**მხოლოდ ერთი გუნდის მიერ**.

ორი გუნდი **არ შეიძლება ერთსა და იმავე bounded context-ზე მუშაობდეს**.

ეს განცალკევება გამორიცხავს იმ **ნაგულისხმევ დაშვებებს**, რომლებიც გუნდებმა შეიძლება ერთმანეთის მოდელებთან დაკავშირებით გააკეთონ.

ამის ნაცვლად გუნდებს უწევთ **მოდელებისა და სისტემების ინტეგრაციისთვის კომუნიკაციის პროტოკოლების მკაფიოდ განსაზღვრა**.

მნიშვნელოვანია აღვნიშნოთ, რომ გუნდებსა და bounded context-ებს შორის ურთიერთობა **ერთმიმართულებიანია**.

ერთი bounded context უნდა ეკუთვნოდეს **მხოლოდ ერთ გუნდს**.

თუმცა **ერთი გუნდი შეიძლება ფლობდეს რამდენიმე bounded context-ს**, როგორც ეს ნაჩვენებია **სურათზე 3-8**.
