**ბიზნეს დომენი შედგება რამდენიმე subdomain-ისგან**.

ამ თავში აქამდე განვიხილეთ ბიზნეს დომენის **უფრო წვრილ პრობლემურ დომენებად ან bounded context-ებად დაშლა**.

თავიდან შეიძლება ჩანდეს, რომ ბიზნეს დომენის დეკომპოზიციის ეს ორი მეთოდი **ერთმანეთის დუბლირებაა**.

თუმცა ეს ასე არ არის.

მოდით ვნახოთ, რატომ გვჭირდება **ორივე საზღვარი**.

---
# Subdomain-ები

კომპანიის ბიზნეს სტრატეგიის გასაგებად საჭიროა **მისი ბიზნეს დომენის ანალიზი**.

Domain-Driven Design მეთოდოლოგიის მიხედვით, ანალიზის ფაზა მოიცავს **სხვადასხვა subdomain-ის იდენტიფიკაციას**:

- core
    
- supporting
    
- generic
    

სწორედ ასე მუშაობს ორგანიზაცია და ასე გეგმავს **თავის კონკურენტულ სტრატეგიას**.

როგორც პირველ თავში ისწავლეთ, **subdomain ჰგავს ერთმანეთთან დაკავშირებული use case-ების ერთობლიობას**.

ეს use case-ები განისაზღვრება:

- ბიზნეს დომენის მიერ
    
- სისტემის მოთხოვნებით
    

როგორც **software engineer-ები**, ჩვენ არ განვსაზღვრავთ მოთხოვნებს  ეს ბიზნესის პასუხისმგებლობაა.

ჩვენი როლია **ბიზნეს დომენის ანალიზი**, რათა გამოვავლინოთ subdomain-ები.

---

# Bounded Context-ები

მეორე მხრივ, **bounded context-ები დიზაინდება**.

მოდელების საზღვრების არჩევა არის **სტრატეგიული დიზაინის გადაწყვეტილება**.

ჩვენ ვწყვეტთ, როგორ გავყოთ ბიზნეს დომენი **უფრო პატარა და მართვად პრობლემურ დომენებად**.


---

# Subdomain-ებისა და Bounded Context-ების ურთიერთქმედება

თეორიულად  თუმცა პრაქტიკაში ნაკლებად რეალისტურად **ერთმა მოდელმა შეიძლება მოიცვას მთელი ბიზნეს დომენი**.

ეს სტრატეგია შეიძლება იმუშაოს პატარა სისტემისთვის, როგორც ნაჩვენებია **სურათზე 3-5**.

როდესაც წარმოიქმნება **კონფლიქტური მოდელები**, შეგვიძლია მივყვეთ დომენის ექსპერტების მენტალურ მოდელებს და **დავშალოთ სისტემა bounded context-ებად**, როგორც ნაჩვენებია **სურათზე 3-6**.

**სურათი 3-6. bounded context-ები, რომლებიც განისაზღვრება ubiquitous language-ის თანმიმდევრულობით**

თუ მოდელები მაინც დიდია და რთულია მათი შენარჩუნება, შეგვიძლია ისინი **კიდევ უფრო პატარა bounded context-ებად დავყოთ**.

მაგალითად, შეგვიძლია გვქონდეს **ერთი bounded context თითო subdomain-ისთვის**, როგორც ნაჩვენებია **სურათზე 3-7**.

![[Pasted image 20260314221821.png]]
**სურათი 3-7. bounded context-ები, რომლებიც align-ებულია subdomain-ების საზღვრებთან**

ნებისმიერ შემთხვევაში ეს არის **დიზაინის გადაწყვეტილება**.
ჩვენ ვქმნით ამ საზღვრებს **როგორც გადაწყვეტის ნაწილს**.

ზოგ შემთხვევაში bounded context-ებსა და subdomain-ებს შორის **ერთი-ერთთან შესაბამისობა** სრულიად გონივრულია.

სხვა შემთხვევებში კი **სხვა დეკომპოზიციის სტრატეგიები შეიძლება უფრო შესაფერისი იყოს**.

ორგანიზაციიდან გამომდინარე, შესაძლოა თქვენ **ორი როლი გქონდეთ** პასუხისმგებელი იყოთ როგორც software engineering-ზე, ისე ბიზნესის განვითარებაზე.

ამ შემთხვევაში თქვენ შეგიძლიათ გავლენა მოახდინოთ როგორც:

- პროგრამული უზრუნველყოფის დიზაინზე (bounded contexts)
    
- ბიზნეს სტრატეგიაზე (subdomains)
    

თუმცა ჩვენი განხილვის კონტექსტში აქ **ფოკუსირებული ვართ მხოლოდ software engineering-ზე**.

მნიშვნელოვანია გვახსოვდეს:

**subdomain-ები აღმოჩენილია (discovered), ხოლო bounded context-ები დაპროექტებულია (designed).**

subdomain-ები განისაზღვრება **ბიზნეს სტრატეგიით**.

მაგრამ ჩვენ შეგვიძლია **დავაპროექტოთ პროგრამული გადაწყვეტა და მისი bounded context-ები**, რათა გავითვალისწინოთ კონკრეტული პროექტის:

- კონტექსტი
    
- შეზღუდვები

ზოგ შემთხვევაში სასარგებლოა **ერთი და იმავე კონცეფციის რამდენიმე მოდელის ერთდროულად გამოყენება**, სხვადასხვა პრობლემის გადასაჭრელად.

როგორც სხვადასხვა ტიპის რუკა გვაწვდის განსხვავებულ ინფორმაციას ჩვენი პლანეტის შესახებ, ასევე შეიძლება გონივრული იყოს **ერთი და იმავე subdomain-ის სხვადასხვა მოდელის გამოყენება სხვადასხვა პრობლემის გადასაჭრელად**.

თუ დიზაინს შევზღუდავთ მხოლოდ **ერთი-ერთთან შესაბამისობით bounded context-ებსა და subdomain-ებს შორის**, ეს შეზღუდავს მოქნილობას და გვაიძულებს გამოვიყენოთ **subdomain-ის მხოლოდ ერთი მოდელი მის bounded context-ში**.
