
Domain-Driven Design-ში გადაწყვეტა მარტივია:  
**საერთო ენა (ubiquitous language) დაიყოს რამდენიმე პატარა ენად**, შემდეგ თითოეულ მათგანს მიენიჭოს **მკაფიო კონტექსტი**, რომელშიც ის შეიძლება გამოიყენებოდეს მისი **bounded context**.

წინა მაგალითში შეგვიძლია გამოვყოთ ორი bounded context: **marketing** და **sales**.

ტერმინი **lead** ორივე bounded context-ში არსებობს, როგორც ეს ნაჩვენებია **სურათზე 3-3**

მანამ, სანამ თითოეულ bounded context-ში მას **ერთი კონკრეტული მნიშვნელობა აქვს**, თითოეული უფრო წვრილად განსაზღვრული ubiquitous language იქნება **თანმიმდევრული** და დაემთხვევა დომენის ექსპერტების **მენტალურ მოდელებს**.

გარკვეული თვალსაზრისით, **ტერმინოლოგიის კონფლიქტები და ნაგულისხმევი კონტექსტები** ნებისმიერი საშუალო ზომის ბიზნესის ბუნებრივი ნაწილია.

Bounded context pattern-ის გამოყენებით ეს კონტექსტები **მოდელირდება როგორც ბიზნეს დომენის აშკარა და განუყოფელი ნაწილი**.

# მოდელის საზღვრები

როგორც წინა თავში განვიხილეთ, **მოდელი არ არის რეალური სამყაროს ასლი**.  
მოდელი არის კონსტრუქცია, რომელიც გვეხმარება **რთული სისტემის გაგებაში**.

პრობლემა, რომლის გადაჭრასაც ის ცდილობს, მოდელის **მნიშვნელოვანი ნაწილი**ა.

მოდელი **ვერ იარსებებს საზღვრის გარეშე**; წინააღმდეგ შემთხვევაში ის გაფართოვდება და საბოლოოდ გახდება რეალური სამყაროს ასლი.

ამიტომ **მოდელის საზღვრის განსაზღვრა მისი bounded context-ები მოდელირების პროცესის განუყოფელი ნაწილია**.

დავუბრუნდეთ რუკების მაგალითს, როგორც მოდელებს.

ჩვენ ვნახეთ, რომ თითოეულ რუკას აქვს თავისი კონკრეტული კონტექსტი:

- საჰაერო რუკა
    
- საზღვაო რუკა
    
- რელიეფის რუკა
    
- მეტროს რუკა
    
- და სხვა
    

რუკა **სასარგებლო და თანმიმდევრულია მხოლოდ მისი კონკრეტული მიზნის ფარგლებში**.

როგორც მეტროს რუკა სრულიად უსარგებლოა **საზღვაო ნავიგაციისთვის**, ასევე ერთი bounded context-ის ubiquitous language შეიძლება **სრულიად შეუსაბამო იყოს სხვა bounded context-ისთვის**.

Bounded context განსაზღვრავს **საერთო ენისა და იმ მოდელის გამოყენების არეს**, რომელსაც ის წარმოადგენს.

ის საშუალებას გვაძლევს განვსაზღვროთ **განსხვავებული მოდელები სხვადასხვა პრობლემური დომენებისთვის**.

სხვა სიტყვებით რომ ვთქვათ:

**Bounded context არის ubiquitous language-ის თანმიმდევრულობის საზღვარი.**

ენის:

- ტერმინოლოგია
    
- პრინციპები
    
- ბიზნეს წესები
    

თანმიმდევრულია **მხოლოდ მისი bounded context-ის შიგნით**.

# დახვეწილი Ubiquitous Language

Bounded context საშუალებას გვაძლევს **დავასრულოთ ubiquitous language-ის განსაზღვრება**.

საერთო ენა **არ არის “საერთო” იმ გაგებით**, რომ ის უნდა გამოიყენებოდეს ორგანიზაციის მთელ მასშტაბზე.

საერთო ენა **არ არის უნივერსალური**.

საერთო ენა **საერთოა მხოლოდ თავისი bounded context-ის საზღვრებში**.

ენა ფოკუსირებულია **მხოლოდ იმ მოდელის აღწერაზე**, რომელიც ამ bounded context-შია მოქცეული.

როგორც მოდელი ვერ იარსებებს იმ პრობლემის გარეშე, რომელსაც ის უნდა გადაჭრას, ასევე **საერთო ენა ვერ განისაზღვრება ან ვერ გამოიყენება მისი გამოყენების მკაფიო კონტექსტის გარეშე**.

# Bounded Context-ის მასშტაბი

თავის დასაწყისში მოცემულმა მაგალითმა აჩვენა ბიზნეს დომენის **ბუნებრივი საზღვარი**.

სხვადასხვა დომენის ექსპერტს ჰქონდა **ერთსა და იმავე ბიზნეს ერთეულზე განსხვავებული მენტალური მოდელები**.

ბიზნეს დომენის მოდელირებისთვის მოგვიწია მოდელის დაყოფა და თითოეული უფრო წვრილი მოდელისთვის **მკაფიო გამოყენების კონტექსტის განსაზღვრა მისი bounded context**.

სართო ენის თანმიმდევრულობა მხოლოდ გვაძლევს საშუალებას განვსაზღვროთ მისი **ყველაზე ფართო საზღვარი**.

ეს საზღვარი ვერ იქნება უფრო დიდი, რადგან მაშინ წარმოიქმნება **არათანმიმდევრული მოდელები და ტერმინოლოგია**.

თუმცა ჩვენ მაინც შეგვიძლია მოდელების **კიდევ უფრო პატარა bounded context-ებად დაყოფა**, როგორც ეს ნაჩვენებია **სურათზე 3-4**.

![[Pasted image 20260314220428.png|697]]
**სურათი 3-4. უფრო პატარა bounded context-ები**

საერთო ენის მასშტაბის განსაზღვრა მისი bounded context-ით არის **სტრატეგიული დიზაინის გადაწყვეტილება**.

საზღვრები შეიძლება იყოს:

- ფართო — ბიზნეს დომენის ბუნებრივი კონტექსტების შესაბამისად
    
- ვიწრო — ბიზნეს დომენის უფრო პატარა პრობლემურ დომენებად დაყოფით

Bounded context-ის ზომა თავისთავად **არ არის გადამწყვეტი ფაქტორი**.
მოდელები არც აუცილებლად დიდი უნდა იყოს და არც პატარა.

**მოდელები უნდა იყოს სასარგებლო.**

რაც უფრო ფართოა ubiquitous language-ის საზღვარი, მით უფრო რთულია მისი **თანმიმდევრულობის შენარჩუნება**.

ხშირად სასარგებლოა დიდი ubiquitous language-ის დაყოფა **უფრო პატარა და მარტივად სამართავ პრობლემურ დომენებად**, მაგრამ ზედმეტად პატარა bounded context-ების შექმნაც შეიძლება პრობლემად იქცეს.

რაც უფრო პატარაა ისინი, მით უფრო **მეტ ინტეგრაციის ხარჯს** იწვევს დიზაინი.

ამიტომ bounded context-ის ზომა უნდა განისაზღვროს **კონკრეტული პრობლემური დომენის მიხედვით**.

ზოგ შემთხვევაში ფართო საზღვარი უფრო ნათელია, ხოლო სხვა შემთხვევებში **დამატებითი დეკომპოზიცია უფრო გონივრულია**.

დიდი bounded context-იდან უფრო წვრილი bounded context-ების გამოყოფის მიზეზები შეიძლება იყოს:

- ახალი **software engineering გუნდის ჩამოყალიბება**
    
- სისტემის **არაფუნქციური მოთხოვნების** შესრულება
    
- კომპონენტების **განვითარების lifecycle-ების განცალკევება**
    

კიდევ ერთი გავრცელებული მიზეზია ფუნქციონალის გამოყოფა იმისთვის, რომ შესაძლებელი იყოს მისი **დამოუკიდებლად მასშტაბირება** სხვა ფუნქციონალებისგან.

ამიტომ:

- შეინარჩუნეთ თქვენი მოდელები 
    
- bounded context-ების ზომები **დაალაგეთ ბიზნესის საჭიროებებსა და ორგანიზაციულ შეზღუდვებთან**

ერთ-ერთი მნიშვნელოვანი საფრთხეა ერთმანეთთან მჭიდროდ დაკავშირებული ფუნქციონალის (High cohesion) რამდენიმე bounded context-ად დაყოფა.

ასეთი დაყოფა გაართულებს თითოეული კონტექსტის **დამოუკიდებელ ევოლუციას**.

ამის ნაცვლად იგივე ბიზნეს მოთხოვნები და ცვლილებები **ერთდროულად იმოქმედებს რამდენიმე bounded context-ზე** და მოითხოვს **ერთდროულ deploy-ს**.

ასეთი არაეფექტური დეკომპოზიციის თავიდან ასაცილებლად გამოიყენეთ **thumb rule**, რომელიც პირველ თავში განვიხილეთ სუბდომენების საპოვნელად:

იპოვეთ **ურთიერთდაკავშირებული use case-ების ჯგუფები**, რომლებიც მუშაობენ ერთსა და იმავე მონაცემებზე და **არ დაყოთ ისინი სხვადასხვა bounded context-ებად**.
