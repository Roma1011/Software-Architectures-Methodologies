სისტემის bounded context-ებს შორის **ინტეგრაციის პატერნების ანალიზის შემდეგ**, შეგვიძლია ისინი **context map-ზე დავხატოთ**, როგორც ნაჩვენებია **სურათზე 4-8**.

![[Pasted image 20260322181844.png]]
**სურათი 4-8. Context map**

**Context map** არის სისტემის **bounded context-ებისა და მათ შორის არსებული ინტეგრაციების ვიზუალური წარმოდგენა**.

ეს ვიზუალური ნოტაცია **მნიშვნელოვან სტრატეგიულ ხედვას იძლევა რამდენიმე დონეზე**:

### მაღალი დონის დიზაინი (High-level design)

Context map გვაძლევს **სისტემის კომპონენტების მიმოხილვას და იმ მოდელებს, რომლებიც ისინი ახორციელებენ**.

### კომუნიკაციის პატერნები (Communication patterns)

Context map აჩვენებს **გუნდებს შორის კომუნიკაციის პატერნებს** მაგალითად:

- რომელი გუნდები თანამშრომლობენ
- და რომელი გუნდები ამჯობინებენ **ნაკლებად მჭიდრო ინტეგრაციის პატერნებს**, როგორიცაა
	- anticorruption layer
	- separate ways
### ორგანიზაციული საკითხები (Organizational issues)

Context map ასევე შეიძლება გვაძლევდეს **ორგანიზაციული პრობლემების შესახებ ინფორმაციას**.

მაგალითად:

რას ნიშნავს, თუ რომელიმე **upstream გუნდის downstream consumers თითქმის ყოველთვის იყენებენ anticorruption layer-ს**,

ან თუ **separate ways პატერნის ყველა იმპლემენტაცია კონცენტრირებულია ერთსა და იმავე გუნდის გარშემო**?

# Maintenance

იდეალურ შემთხვევაში **context map პროექტში თავიდანვე უნდა დაინერგოს**, და უნდა **განახლდეს**, რათა ასახავდეს:

- ახალი bounded context-ების დამატებას
- უკვე არსებული bounded context-ების ცვლილებებს.

რადგან context map შესაძლოა შეიცავდეს **რამდენიმე გუნდის მუშაობიდან მომდინარე ინფორმაციას**, უმჯობესია context map-ის მოვლა **გაზიარებულ პასუხისმგებლობად განისაზღვროს**.

თითოეული გუნდი პასუხისმგებელია **საკუთარი ინტეგრაციების განახლებაზე სხვა bounded context-ებთან**.

context map-ის მართვა და შენარჩუნება შესაძლებელია **როგორც კოდის**, ისეთი ინსტრუმენტის გამოყენებით, როგორიც არის **Context Mapper**.****

# შეზღუდვები (Limitations)

მნიშვნელოვანია აღვნიშნოთ, რომ **context map-ის აგება შეიძლება საკმაოდ რთული ამოცანა იყოს**.

როდესაც სისტემის bounded context-ები მოიცავს **რამდენიმე subdomain-ს**, შეიძლება არსებობდეს **რამდენიმე სხვადასხვა ინტეგრაციის პატერნი ერთდროულად**.

მაგალითად, **სურათზე 4-9** შეგიძლიათ იხილოთ ორი bounded context, რომელთა შორის გამოიყენება **ორი განსხვავებული ინტეგრაციის პატერნი**:

- partnership
- anticorruption layer

![[Pasted image 20260322182405.png]]
**სურათი 4-9. რთული context map**

გარდა ამისა, მაშინაც კი თუ bounded context-ები **შეზღუდულია მხოლოდ ერთი subdomain-ით**, მაინც შეიძლება არსებობდეს **რამდენიმე ინტეგრაციის პატერნი ერთდროულად**.

მაგალითად, როდესაც **subdomain-ის სხვადასხვა მოდული განსხვავებულ ინტეგრაციის სტრატეგიას მოითხოვს**.
