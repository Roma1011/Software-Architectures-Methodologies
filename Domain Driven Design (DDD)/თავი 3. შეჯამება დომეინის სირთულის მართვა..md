როდესაც დომენის ექსპერტების **მენტალურ მოდელებში არსებულ კონფლიქტს** ვაწყდებით, საჭიროა **საერთო ენის დაყოფა რამდენიმე bounded context-ად**.

საერთო ენა უნდა იყოს **თანმიმდევრული თავისი bounded context-ის ფარგლებში**.  
თუმცა სხვადასხვა bounded context-ებს შორის **ერთსა და იმავე ტერმინებს შეიძლება განსხვავებული მნიშვნელობა ჰქონდეს**.

მიუხედავად იმისა, რომ **subdomain-ები აღმოჩენილია (discovered)**, **bounded context-ები დაპროექტებულია (designed)**.  
დომენის bounded context-ებად დაყოფა არის **სტრატეგიული დიზაინის გადაწყვეტილება**.

ერთი bounded context და მისი ubiquitous language შეიძლება **იმპლემენტირებული და შენარჩუნებული იყოს ერთი გუნდის მიერ**.

ორი გუნდი **არ შეიძლება ერთსა და იმავე bounded context-ზე ერთად მუშაობდეს**.  
თუმცა **ერთ გუნდს შეუძლია რამდენიმე bounded context-ზე მუშაობა**.

Bounded context-ები სისტემას ანაწილებენ **ფიზიკურ კომპონენტებად**, მაგალითად:

- სერვისებად
    
- ქვესისტემებად
    
- და სხვა მსგავს კომპონენტებად.
    

თითოეული bounded context-ის **ცხოვრების ციკლი განცალკევებულია დანარჩენი სისტემისგან**.

თითოეული bounded context შეიძლება **დამოუკიდებლად განვითარდეს სისტემის სხვა ნაწილებისგან**.

თუმცა bounded context-ებს მაინც უნდა შეეძლოთ **ერთად მუშაობა სისტემის შესაქმნელად**.

ზოგიერთი ცვლილება **აუცილებლად იმოქმედებს სხვა bounded context-ებზეც**.

შემდეგ თავში ვისაუბრებთ **bounded context-ების ინტეგრაციის სხვადასხვა პატერნზე**, რომლებიც შეიძლება გამოყენებულ იქნას **მათ cascading ცვლილებებისგან დასაცავად**.
