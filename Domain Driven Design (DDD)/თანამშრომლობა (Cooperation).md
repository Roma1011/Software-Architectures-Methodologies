**Cooperation პატერნები** ეხება იმ bounded context-ებს, რომლებიც იმპლემენტირებულია გუნდების მიერ, რომელთაც შორის **კარგად ჩამოყალიბებული კომუნიკაცია არსებობს**.

ყველაზე მარტივ შემთხვევაში ეს არის bounded context-ები, რომლებიც **ერთი გუნდის მიერ არის იმპლემენტირებული**.

ეს ასევე ეხება ისეთ გუნდებსაც, რომელთა **მიზნები ერთმანეთზეა დამოკიდებული**, როდესაც ერთი გუნდის წარმატება დამოკიდებულია მეორე გუნდის წარმატებაზე და პირიქით.

ამ შემთხვევაშიც მთავარი კრიტერიუმია **გუნდებს შორის კომუნიკაციისა და თანამშრომლობის ხარისხი**.

---

### Partnership

**Partnership მოდელში** bounded context-ებს შორის ინტეგრაცია **ad hoc წესით კოორდინირდება**.

ერთ გუნდს შეუძლია მეორე გუნდს აცნობოს **API-ში განხორციელებული ცვლილების შესახებ**, ხოლო მეორე გუნდი ითანამშრომლებს და მოერგება ამ ცვლილებას  **კონფლიქტების გარეშე** (იხილეთ სურათი 4-1).

![[Pasted image 20260322165209.png]]
**სურათი 4-1. Partnership მოდელი**

ამ შემთხვევაში ინტეგრაციის კოორდინაცია **ორმხრივია**.

არც ერთი გუნდი **არ კარნახობს ენას**, რომელიც გამოიყენება contract-ების განსაზღვრისთვის.

გუნდებს შეუძლიათ **განსხვავებების შეთანხმებით გადაჭრა** და **ყველაზე შესაფერისი გადაწყვეტის არჩევა**.

ასევე, ორივე მხარე თანამშრომლობს **ნებისმიერი ინტეგრაციის პრობლემის გადაწყვეტაში**, რომელიც შეიძლება წარმოიშვას.

არც ერთ გუნდს **არ აქვს ინტერესი მეორე გუნდის მუშაობის დაბლოკვაში**.

ამგვარი ინტეგრაციის წარმატებისთვის საჭიროა:

- კარგად ჩამოყალიბებული თანამშრომლობის პრაქტიკები
    
- მაღალი დონით პასუხისმგებლობა (commitment)
    
- გუნდებს შორის ხშირი სინქრონიზაცია
    

ტექნიკური თვალსაზრისით საჭიროა **ორივე გუნდის მიერ განხორციელებული ცვლილებების უწყვეტი ინტეგრაცია (continuous integration)**, რათა კიდევ უფრო შემცირდეს **ინტეგრაციის feedback loop**.

ეს პატერნი შეიძლება **არ იყოს კარგი არჩევანი გეოგრაფიულად განაწილებული გუნდებისთვის**, რადგან ასეთ შემთხვევაში შეიძლება წარმოიშვას **სინქრონიზაციისა და კომუნიკაციის სირთულეები**.

---
### საერთო ბირთვი (Shared Kernel)

მიუხედავად იმისა, რომ **bounded context-ები მოდელის საზღვრებია**, მაინც შეიძლება არსებობდეს შემთხვევები, როდესაც **subdomain-ის იგივე მოდელი, ან მისი ნაწილი**, იმპლემენტირებული იქნება **რამდენიმე bounded context-ში**.

მნიშვნელოვანია ხაზგასმით აღვნიშნოთ, რომ **გაზიარებული მოდელი დაპროექტებულია ყველა bounded context-ის საჭიროებების მიხედვით**.

უფრო მეტიც, **გაზიარებული მოდელი თანმიმდევრული უნდა იყოს ყველა იმ bounded context-ში, რომელიც მას იყენებს**.

მაგალითად, განვიხილოთ **Enterprise სისტემა**, რომელიც იყენებს სპეციალურად შექმნილ მოდელს **მომხმარებლების ნებართვების (permissions) მართვისთვის**.

თითოეულ მომხმარებელს შეიძლება ჰქონდეს:

- ნებართვები მინიჭებული პირდაპირ
- ან მემკვიდრეობით მიღებული იმ **ორგანიზაციული ერთეულიდან**, რომელსაც ის ეკუთვნის.

უფრო მეტიც, **თითოეულ bounded context-ს შეუძლია ავტორიზაციის მოდელის შეცვლა**, და ის ცვლილებები, რომლებიც თითოეული bounded context-ის მიერ განხორციელდება, უნდა **აისახოს ყველა სხვა bounded context-ზე**, რომელიც ამ მოდელს იყენებს (იხილეთ სურათი 4-2).

# გაზიარებული არე (Shared Scope)

გადაფარული მოდელი **აკავშირებს მონაწილე bounded context-ების სიცოცხლის ციკლებს**.

გაზიარებულ მოდელში განხორციელებული ცვლილება **მყისიერად მოქმედებს ყველა bounded context-ზე**.

ამიტომ, ცვლილებების **კასკადური ეფექტების მინიმიზაციისთვის**, გადაფარული მოდელი უნდა იყოს **შეზღუდული**, და უნდა ავლენდეს მხოლოდ იმ ნაწილს, რომელიც ორივე bounded context-ის მიერ უნდა იყოს იმპლემენტირებული.

იდეალურ შემთხვევაში, **shared kernel** უნდა შედგებოდეს მხოლოდ:

- ინტეგრაციის contract-ებისგან
- მონაცემთა სტრუქტურებისგან

რომლებიც განკუთვნილია **bounded context-ების საზღვრებს შორის გადასაცემად**.

# იმპლემენტაცია (Implementation)

**Shared kernel იმპლემენტირებულია ისე**, რომ მის source code-ში ნებისმიერი ცვლილება **მყისიერად აისახება ყველა იმ bounded context-ში, რომელიც მას იყენებს**.

თუ ორგანიზაცია იყენებს **mono-repository მიდგომას**, ეს შეიძლება იყოს იგივე source ფაილები, რომლებსაც რამდენიმე bounded context ერთდროულად იყენებს.

თუ **გაზიარებული repository-ის გამოყენება შეუძლებელია**, shared kernel შეიძლება **გამოიყოს ცალკე პროექტად** და bounded context-ებში იყოს გამოყენებული **როგორც დაკავშირებული ბიბლიოთეკა (linked library)**.

ნებისმიერ შემთხვევაში, **shared kernel-ში განხორციელებულმა თითოეულმა ცვლილებამ უნდა გამოიწვიოს ინტეგრაციის ტესტების გაშვება ყველა იმ bounded context-ისთვის, რომელიც ამ ცვლილებით ზიანდება**.

ცვლილებების **უწყვეტი ინტეგრაცია აუცილებელია**, რადგან shared kernel ეკუთვნის **რამდენიმე bounded context-ს**.

თუ shared kernel-ში ცვლილებები **არ გავრცელდება ყველა შესაბამის bounded context-ზე**, ეს გამოიწვევს **მოდელის არათანმიმდევრულობას**.

ასეთ შემთხვევაში bounded context-ები შეიძლება დაეყრდნონ **shared kernel-ის მოძველებულ იმპლემენტაციას**, რაც გამოიწვევს:

- მონაცემთა დაზიანებას (data corruption)
- და/ან runtime პრობლემებს.

# როდის უნდა გამოვიყენოთ Shared Kernel

Shared kernel პატერნის გამოყენების ძირითადი კრიტერიუმია:

**დუბლირების ღირებულება(duplication) vs კოორდინაციის ღირებულება(coordination.)**.

რადგან ეს პატერნი ქმნის **ძლიერ დამოკიდებულებას მონაწილე bounded context-ებს შორის**, ის უნდა იქნას გამოყენებული მხოლოდ მაშინ, როდესაც **დუბლირების ღირებულება უფრო მაღალია, ვიდრე კოორდინაციის ღირებულება**.

სხვა სიტყვებით, მხოლოდ მაშინ, როდესაც **ორივე bounded context-ის მიერ გაზიარებულ მოდელში განხორციელებული ცვლილებების ინტეგრაცია მოითხოვს უფრო მეტ ძალისხმევას, ვიდრე ცვლილებების კოორდინაცია გაზიარებულ codebase-ში**.

ინტეგრაციისა და დუბლირების ღირებულებებს შორის განსხვავება დამოკიდებულია **მოდელის ცვალებადობაზე (volatility)**.

რაც უფრო ხშირად იცვლება მოდელი, მით უფრო მაღალი იქნება **ინტეგრაციის ხარჯი**.

ამიტომ shared kernel ბუნებრივად გამოიყენება იმ **subdomain-ებისთვის, რომლებიც ყველაზე ხშირად იცვლება core subdomain-ებისთვის**.


გარკვეული თვალსაზრისით, **shared kernel პატერნი ეწინააღმდეგება bounded context-ების პრინციპებს**, რომლებიც წინა თავში განვიხილეთ.

თუ მონაწილე bounded context-ები **ერთი და იმავე გუნდის მიერ არ არის იმპლემენტირებული**, shared kernel-ის დანერგვა ეწინააღმდეგება პრინციპს, რომლის მიხედვითაც **ერთ bounded context-ს უნდა ფლობდეს ერთი გუნდი**.

გაზიარებული მოდელი  shared kernel  ფაქტობრივად **რამდენიმე გუნდის მიერ ვითარდება**.

სწორედ ამიტომ shared kernel-ის გამოყენება **უნდა იყოს კარგად დასაბუთებული**.

ეს არის **პრაგმატული გამონაკლისი**, რომელიც უნდა იქნას გამოყენებული ფრთხილად.

shared kernel-ის გამოყენების ერთ-ერთი გავრცელებული შემთხვევაა, როდესაც **კომუნიკაციის ან თანამშრომლობის პრობლემები ხელს უშლის partnership pattern ის გამოყენებას**, მაგალითად:

- გეოგრაფიული შეზღუდვების გამო
- ან ორგანიზაციული პოლიტიკის გამო.

თუ მჭიდროდ დაკავშირებული ფუნქციონალი იმპლემენტირდება **სათანადო კოორდინაციის გარეშე**, ეს გამოიწვევს:

- ინტეგრაციის პრობლემებს
- დესინქრონიზებულ მოდელებს
- კამათს იმის შესახებ, თუ რომელი მოდელია უკეთ დაპროექტებული.

shared kernel-ის მასშტაბის მინიმიზაცია **აკონტროლებს კასკადური ცვლილებების მასშტაბს**, ხოლო **ყოველი ცვლილებისას ინტეგრაციის ტესტების გაშვება უზრუნველყოფს ინტეგრაციის პრობლემების ადრეულ აღმოჩენას**.

---

shared kernel პატერნის გამოყენების კიდევ ერთი გავრცელებული შემთხვევა თუმცა დროებითი არის **legacy სისტემის ეტაპობრივი მოდერნიზაცია**.

ასეთ სცენარში **გაზიარებული codebase შეიძლება იყოს პრაგმატული შუალედური გადაწყვეტა**, რომელიც სისტემის bounded context-ებად ეტაპობრივ დაშლას უზრუნველყოფს.


---

ბოლოს, shared kernel შეიძლება კარგი არჩევანი იყოს **იმ bounded context-ების ინტეგრაციისთვის, რომლებიც ერთსა და იმავე გუნდის მიერ ფლობს და იმპლემენტირდება**.

ასეთ შემთხვევაში bounded context-ების **ad hoc ინტეგრაციამ  partnership-მა დროთა განმავლობაში შეიძლება გააქროს კონტექსტების საზღვრები**.

shared kernel შეიძლება გამოყენებულ იქნას **bounded context-ებს შორის ინტეგრაციის contract-ების მკაფიოდ განსაზღვრისთვის**.
